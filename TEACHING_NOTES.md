# CORDY Interest Profiler — Teaching Notes

A running log of the coding principles, architecture decisions, and systems concepts covered while building this app. Organized by file so you can jump to whichever part you're working on.

## How to run this app

```bash
cd /Users/jay/Documents/Cordy/Dev/Interest_Profiler
npm run dev
```

Open `http://localhost:3000`.

- `npm run dev` — starts the **development** server. Watches files, hot-reloads on save, gives readable error messages with source maps. Use this while building.
- `npm run build && npm start` (or `npm run preview`) — builds an **optimized production bundle** (minified, no hot-reload, faster runtime) and serves it. This is what you'd deploy to a real server — never run `dev` in production.
- `npm run typecheck` — runs TypeScript's compiler in check-only mode (`tsc --noEmit`) to catch type errors without producing output files.
- `npm run lint` — runs ESLint to catch code-quality and correctness issues beyond what TypeScript checks.

## `src/lib/types.ts` — Shared type contracts

**Separation of concerns.** Types live in their own file, imported by both the client (chat page) and server (API route). This means the "shape" of a message or profile is defined once — if it changes, TypeScript will flag every place that needs updating.

**`Omit<Message, "id">` for `ChatRequest`.** The client tracks messages with an `id` for React's list rendering (`key={msg.id}`), but the Claude API doesn't need or want that field — it only cares about `role` and `content`. `Omit` lets us derive a narrower type from a wider one instead of maintaining two separate, and possibly-diverging, definitions.

**Optional fields (`?`) signal "this might not exist yet."** `Opportunity.url` is optional because Claude-generated fallback opportunities have no real link. `ProfileData._opportunityQueries` is optional and prefixed with `_` to signal "internal, temporary, not part of the public contract" — it's a value smuggled from `extractProfile` to the API route, then deleted before the response is sent to the browser.

## `src/lib/prompts.ts` — Prompt engineering & parsing boundaries

**Offload judgment to the model, keep code thin.** Instead of writing a state machine to track "how many questions have been asked, what topics are covered," all of that logic lives in the system prompt as plain English instructions. The model handles conversational judgment; the code just ferries messages back and forth. This is a core mindset shift when building on LLMs — you're not writing the control flow, you're writing instructions for something that already has control flow.

**Parse at a single boundary.** `extractProfile` and `stripProfileBlock` are the *only* two places in the whole app that know the `<profile>...</profile>` XML-tag format Claude's output uses. If that format ever needs to change, there's exactly one place to fix it — this is the **single source of truth** principle applied to data parsing, not just data storage.

**Regex as a lightweight parser.** `/<profile>\s*([\s\S]*?)\s*<\/profile>/` finds the block; `JSON.parse` on the captured group turns it into a real object. Wrapping the `JSON.parse` in try/catch matters because we're parsing text generated by a language model — it's usually well-formed, but "usually" isn't "always," so a malformed block shouldn't crash the whole request.

## `src/app/api/chat/route.ts` — Server boundary & Next.js API routes

**The security boundary — why this route exists at all.** The Anthropic API key must never reach the browser. If the client called Claude directly, the key would have to be embedded in JavaScript sent to every visitor — trivially stealable via DevTools. The API route runs on the server; the client only ever talks to `/api/chat`, and the server adds the secret. This server-side proxy pattern is called a **backend-for-frontend (BFF)**. Any credential — API keys, database passwords, payment tokens — lives on the server, never the client.

**App Router's route convention.** Files named `route.ts` export functions named after HTTP verbs (`GET`, `POST`, etc.), each receiving a Web-standard `Request` and returning a `Response`. This differs from the older Pages Router (`pages/api/*.ts` with a single default export) — App Router uses Web standard APIs so routes can run at the edge, not just in Node.js.

**Validate at the boundary, trust the interior.** Anything crossing from outside your system — HTTP request bodies, third-party API responses — gets checked before use. A TypeScript type cast (`as ChatRequest`) is a compile-time-only promise; it does nothing at runtime. The actual guard is the manual `Array.isArray` / length check. Inside your own code, calling your own functions, you don't re-validate everywhere — validation happens at the edges.

**Module-level singleton.** `const anthropic = new Anthropic(...)` sits outside the `POST` function, at module scope. It's created once when the server process starts and reused across every request. Creating it inside `POST` would mean re-instantiating the SDK client on every single call — wasteful, since it's stateless after construction. Rule of thumb: expensive-to-create + stateless = initialize once at module level (same pattern used for DB connection pools, loggers, HTTP clients).

**HTTP status codes communicate *who's* at fault.** `400` = bad request (client's fault, fix your data). `502` = bad gateway (we tried to call Claude and it failed — our fault, but not a code bug). Returning `200` with `{ error: "..." }` in the body is a common anti-pattern — it breaks HTTP-aware tooling that assumes 2xx means success.

**Log verbosely server-side, respond minimally client-side.** `console.error` captures the real error (stack trace, details) for debugging; the client only gets a safe, generic message. Exposing internal error details to callers is a security leak (it can reveal internal service names, data shapes, or stack traces attackers can use).

**`Promise.all` for parallel async work.** Fetching multiple opportunity queries with a `for` loop and `await` inside would run them one after another — three sequential ~200ms fetches becomes ~600ms. `Promise.all(queries.map(fetch...))` fires all requests simultaneously and waits for all to settle — same total time as the slowest single request. Tradeoff: if any promise rejects, `Promise.all` rejects immediately, so it needs a surrounding try/catch (here, falling back to generated opportunities on failure).

## `src/app/chat/page.tsx` — React state, effects, and client/server split

**`"use client"` — the two rendering worlds of App Router.** By default every file in App Router is a **Server Component**: it renders on the server and ships zero JavaScript to the browser. React hooks (`useState`, `useEffect`, etc.) only work in the browser, because they manage state living in browser memory — so any component using them needs `"use client"` to opt into the client JavaScript bundle. Rule of thumb: push `"use client"` as far down the component tree as possible, so only the interactive parts ship JS. Here, the whole chat screen is inherently interactive, so the whole page is a client component.

**`useState` is React's memory across re-renders.** A component is just a function that re-runs top-to-bottom whenever something changes. A plain variable would reset to its initial value every re-render. `useState` persists a value between renders and gives you a setter that, when called, schedules React to re-render with the new value.

**Derived state — one source of truth.** `isLoading` isn't its own `useState` — it's computed each render as `status !== "chatting"`. If you store two variables that need to stay in sync manually, you'll eventually forget to update one of them. Compute what you can from the minimal state you actually need.

**`useRef` — mutable values that don't trigger renders.** Unlike `useState`, writing to a ref does not cause a re-render. Used here for (1) DOM references — `bottomRef` points to an invisible div so `scrollIntoView()` can be called on it directly, since React doesn't manage scroll position as state, and (2) imperative actions — `inputRef.current?.focus()` to restore focus after a reply, a side effect outside React's normal render flow.

**`useEffect` — synchronizing with things outside React's render.** Scrolling isn't something you can express as "what does the UI look like given this state" — it's an imperative action that needs to happen *after* React updates the DOM. `useEffect(fn, [messages, status])` re-runs `fn` whenever `messages` or `status` changes since the last render. The dependency array must include every value the effect reads — omitting one causes stale-closure bugs where the effect keeps using an old value.

**Async state updates aren't immediate — the `nextMessages` pattern.** `setMessages(nextMessages)` doesn't synchronously update `messages` — React batches the update for the next render. So the code builds `nextMessages` as a local variable first and sends *that* to the API, rather than reading the (still-stale) `messages` variable right after calling the setter. This is one of the most common React footguns.

**New references, not mutations.** `[...messages, userMessage]` creates a new array. React detects changes by comparing references (`===`), not deep equality — mutating the existing array in place (`messages.push(...)`) keeps the same reference, and React may not notice the change at all. Always create new arrays/objects when updating state.

**Guard clauses keep the happy path readable.** `if (!content.trim() || isLoading) return;` at the top of a function, rather than wrapping the entire body in `if (...) { ... }`, keeps the main logic unindented and easy to scan.

**`localStorage` as a simple cross-page data bridge.** With no database and no global state library, the extracted profile is serialized with `JSON.stringify` and stored under a key in `localStorage`, then read back with `JSON.parse` on the profile page. Alternatives considered: URL query params (profiles can get large; sensitive data shouldn't sit in browser history/server logs) and a state management library (unnecessary complexity for a one-time handoff between two pages).

**Component composition — single responsibility per component.** `ChatPage` doesn't know how to draw a message bubble or animate typing dots — it delegates to `ChatBubble` and `TypingIndicator`, passing data down as props. Each component does one job. Change the bubble's visual design without touching the conversation logic, and vice versa.

**`void` before a fire-and-forget async call.** `onSelect={(r) => void sendMessage(r)}` — event handlers are expected to return nothing, but `sendMessage` returns a Promise. The `void` operator explicitly tells TypeScript "I'm intentionally not awaiting or handling this promise here" (the function's own try/catch handles errors internally), rather than silently ignoring an unhandled-promise warning.

## Rebrand to match CORDY's real design system

Once we had reference screenshots of the actual cordy.sg site, we threw out the placeholder dark theme and rebuilt the visual system to match their real brand: cream background, red mascot/logo, teal action buttons, thick black borders on every card, and a chunky rounded heading font. A few concepts came out of that process worth remembering for any future rebrand or design-system work.

**Tailwind v4's `@theme` block generates utility classes for you.** In [globals.css](src/styles/globals.css):

```css
@theme {
  --color-cordy-red: #ef4444;
  --color-cordy-teal: #2dd4a7;
  --color-cordy-ink: #16213e;
}
```

Any CSS variable named `--color-{name}` inside `@theme` automatically becomes usable as `bg-{name}`, `text-{name}`, `border-{name}`, etc. — here, `bg-cordy-red`, `border-cordy-ink`. This is new in Tailwind v4; in v3 you'd do this in a separate `tailwind.config.js` file under `theme.extend.colors`. The payoff: colors are defined **once**, and every component references the semantic name (`cordy-red`) instead of repeating a hex code (`#EF4444`) everywhere. If the brand red ever changes, there's one line to edit.

**Why a semantic name instead of the raw hex, even for a one-off brand.** `bg-[#EF4444]` works fine in Tailwind (arbitrary values), but it means the color is just a magic number scattered across a dozen files. `bg-cordy-red` documents *intent* — "this is our brand red" — and gives you a single source of truth to change later. This is the same "single source of truth" principle from earlier files, applied to design tokens instead of data types.

**Loading custom Google Fonts in Next.js.** `next/font/google` (used in [layout.tsx](src/app/layout.tsx)) downloads and self-hosts the font at build time — no external request to Google's CDN at runtime, which avoids a layout-shift-causing render-blocking request and keeps the page fast. Each font is given a CSS variable name (`--font-baloo`, `--font-nunito`) via the `variable` option, which is how `globals.css` references them in the `@theme` block without ever needing to know the font's actual family name string.

**Two fonts, two jobs.** `Baloo 2` (chunky, rounded, high personality) is used only for headings via a `font-heading` utility; `Nunito` (a clean, readable sans) handles all body text. This is a common typography pattern: a distinctive display font draws attention at a few key moments (titles, CTAs), while a neutral font carries the bulk of the reading text where personality would become fatiguing over a long paragraph.

**The neo-brutalist "sticker" shadow.** CORDY's cards use a hard-edged offset shadow with no blur — e.g. `shadow-[3px_3px_0_0_var(--color-cordy-ink)]` — paired with a solid `border-2`. Compare this to a normal soft box-shadow (`shadow-lg`), which is blurred and meant to look like ambient light. A zero-blur, solid-color offset shadow instead mimics a sticker or pop-art panel sitting slightly off the page — it's a deliberate, chunky, playful look rather than a realistic lighting effect. Tailwind's arbitrary value syntax (`shadow-[...]`) lets you drop in a raw CSS `box-shadow` value when the preset `shadow-sm/md/lg` scale doesn't match what you need.

## Building the welcome and profile pages — new patterns

**Reading from `localStorage` needs `useEffect`, not a direct read during render.** In [profile/page.tsx](src/app/profile/page.tsx):

```tsx
const [profile, setProfile] = useState<ProfileData | null>(null);
useEffect(() => {
  const raw = localStorage.getItem("cordy_profile");
  if (raw) setProfile(JSON.parse(raw) as ProfileData);
  setLoaded(true);
}, []);
```

You might expect `localStorage.getItem(...)` could just run directly in the component body. Two problems with that: first, `localStorage` doesn't exist during server-side rendering (Next.js renders this component's initial HTML on the server, where there's no browser and no `localStorage` — reading it directly would crash). Second, even client-side, reading storage during render is a **side effect** (same category as the scrolling/focus effects from the chat page) — the convention is that rendering should be a pure function of props/state, and anything that reaches outside the component (browser APIs, network, timers) belongs in `useEffect`, which only runs after the component has mounted in the browser.

**The empty dependency array `[]` means "run once, on mount."** Unlike the chat page's `useEffect(fn, [messages, status])` which re-runs whenever those values change, `useEffect(fn, [])` runs exactly once, right after the first render. That's correct here because we only need to check `localStorage` a single time when the page loads — there's nothing to keep re-syncing with afterward.

**A `loaded` flag to distinguish "haven't checked yet" from "checked and found nothing."** Without it, `profile === null` is ambiguous: it could mean "we haven't read localStorage yet" or "we read it and there's genuinely no profile." The `loaded` boolean disambiguates those two states so we don't flash a "no profile found" error message for a split second before the real data arrives. This is a common pattern any time you have an async or effect-driven data load — plain `null` isn't enough state to represent "loading vs. empty vs. loaded."

**Guard clauses for page-level states, not just function-level ones.** The profile page has three possible states — not loaded yet, loaded but empty, loaded with data — handled as sequential early returns (`if (loaded && !profile) return <EmptyState/>`) rather than one giant nested ternary in the JSX. Same principle as the guard clauses in `sendMessage` on the chat page: keep the "main" rendering path un-nested and easy to read by handling edge cases first and returning early.

**`next/link` vs. a plain `<a>` tag.** Both the welcome and profile pages use `<Link href="/chat">` from `next/link` instead of `<a href="/chat">`. Next.js's `Link` component intercepts the navigation and does a **client-side transition** — it swaps content without a full page reload, reusing already-downloaded JavaScript. A plain `<a>` tag would force a full browser navigation (reload every script, re-run all setup) even though we're staying within the same app. Always use `Link` for internal navigation within a Next.js app; reserve plain `<a>` for external URLs.

## Debugging a real layout bug — a worked example

While checking the welcome page in the browser preview, the hero button and heading appeared clipped at the edges of the screen. Here's the actual debugging process, since it's a good template for diagnosing CSS layout bugs generally:

1. **Don't guess — inspect computed values.** Rather than assuming what's wrong from the screenshot alone, `preview_inspect` was used to check the *actual rendered* CSS of the `<h1>` (`width`, `max-width`, `font-size`) and compare that to what was expected. The screenshot shows the visual symptom; computed styles show the mechanism.

2. **Check for horizontal overflow specifically.** `document.body.scrollWidth` vs. `window.innerWidth` is the standard test for "is something wider than the viewport forcing an unwanted horizontal scrollbar?" If `scrollWidth > innerWidth`, something is overflowing and the browser is auto-scrolling to it, which is exactly what made the heading look "cut off on both sides" — it wasn't actually clipped, the viewport had scrolled sideways to center on a wider sibling element.

3. **Find the actual culprit, not just the symptom.** The real cause was `whitespace-nowrap` on the CTA button — a class that forces text to stay on one line no matter how narrow the container is. At a narrow mobile width, "Start chatting with CORDY →" as one unbroken line is simply wider than the screen, so the button (and the whole page, since it doesn't wrap) became wider than the viewport. The fix wasn't to shrink font sizes or add more padding rules — it was to remove the one class that was preventing natural text wrapping. **Lesson: `whitespace-nowrap` is only safe on short, fixed-length labels (like "SIGN UP") — never on a full sentence CTA that needs to adapt across screen sizes.**

4. **Verify the fix at a realistic viewport, not just the tool's default.** The preview tool's default screenshot viewport turned out to be unusually narrow (~198 CSS px) — narrower than any real phone. After confirming the fix, the viewport was explicitly resized to a real device size (390×844, an iPhone-class width) before calling the layout "verified," since a fix that only looks right at an artificial 198px width isn't actually proven to work on a real device.
